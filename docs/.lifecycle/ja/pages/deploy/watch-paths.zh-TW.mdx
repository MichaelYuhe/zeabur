# トリガーパス

サービスのデプロイが完了すると、コードを頻繁に更新する必要がある場合がありますが、すべてのコードの更新がサービスの再デプロイを必要としないこともあります。

たとえば、フロントエンドとバックエンドのコードを同じGitリポジトリに配置したプロジェクトでは、バックエンドの一部のコードを更新しただけでも、コミット後にフロントエンドのサービスも再デプロイされることがあります。

これを避けたい場合は、トリガーパスの設定を変更するだけで済みます：

### ステップ1：サービス設定ページを開く

構成を変更したいサービスを選択し、設定ページを展開し、下にスクロールして、**監視パス**項目を見つけてサービスの監視パスを変更することができます。

![setting](/deploy/watch-path/setting.zh-TW.png)

### ステップ2：トリガーパスの変更

デフォルトのトリガーパスは `*` であり、これはサービス全体のリポジトリをトリガーします。リポジトリ内の任意のファイルの更新によってサービスが再デプロイされます。

`.gitignore` ファイルの書式に従って、カスタムのトリガーパスを更新できます。

ただし、`.gitignore` は無視を表していますが、ここでのトリガーパスはトリガーを表しています。

たとえば、フロントエンドとバックエンドを一体化したアプリケーションをデプロイした場合、次のようにコードがフォーマットされます。

![repo](/deploy/watch-path/repo.png)

このリポジトリには `client` ルートを持つ1つのサービスと `server` ルートを持つ別のサービスがデプロイされていますが、これはビルドパスを指定しただけであり、さらに隔離を行うには **Watch paths** を設定する必要があります。

たとえば、クライアントサービスの **Watch paths** を `/client` に設定すると、`server` フォルダーで行った任意の更新は自動的にクライアントサービスの再デプロイをトリガーしません。
